## Context

The catalog-service was initially built around a single flat model, `LeatherProduct`,
that described a leather goods item as a single row: name, description, price,
color, size, and a `stock_quantity` integer column directly on the product record.
The full stack for this model was in place: a domain type, a migration, a
repository, a service layer, HTTP handlers, and seed data.

As the design evolved toward a more realistic e-commerce model, a second domain
model was introduced: `Product` with a separate `ProductVariant` child table.
This normalized shape supports multiple SKUs per product (different sizes and
colors), each with its own pricing and stock. A second full stack was built
alongside the first: new domain types, new repository interfaces and PostgreSQL
implementations, new service interfaces and default implementations, new HTTP
handlers, a second migration, and a second seed file.

The `LeatherProduct` stack was never removed.

At the same time, `inventory-service` was introduced as a separate gRPC service
responsible for authoritative stock management, with its own PostgreSQL database
and transactional reservation logic.

## Hypothesis

The implicit belief was that both domain models could coexist temporarily while
the second one was being built out, and that the first would be removed "later"
once the second was complete. The assumption was that parallel stacks were a
reasonable scaffolding strategy for an incremental migration.

## Failure

By the time an integrity analysis was performed, the system had **two complete,
live domain stacks** in the same service:

| Layer | v1 (LeatherProduct) | v2 (Product / ProductVariant) |
|---|---|---|
| Domain type | `LeatherProduct` | `Product`, `ProductVariant`, `ProductImage` |
| Migration | `001_create_tables.sql` (`products` table) | `002_create_catalog_tables.sql` (`catalog_products`, `product_variants`, `product_images`) |
| Repository | `LeatherProductRepository` | `CatalogProductRepository`, `ProductVariantRepository`, `ProductImageRepository`, `CatalogSyncRepository` |
| Service | `LeatherProductService` | `CatalogService`, `ProductService` |
| Handler | `LeatherProductHandler` | `CatalogHandler`, `ProductHandler` |
| Seed | `seed.Run` populating `products` | `seed.Run` populating `catalog_products` + `product_variants` |

Beyond the structural duplication, there were **three concurrent sources of
truth for stock quantity**:

1. `products.stock_quantity` â€” the integer column on the v1 flat product table.
2. `product_variants.stock_quantity` â€” the column on each variant row in the v2
   normalized table.
3. `inventory-service` â€” the authoritative gRPC service with transactional
   reservation semantics.

All three could disagree at any moment. The gateway was wired to derive stock
status from inventory-service (the correct source), but both catalog tables
still carried stock columns with stale or seed-only values.

## Why it went undetected

The failure was structurally invisible to every standard quality signal:

- **Compilation:** Both stacks were syntactically valid Go. No type errors, no
  unused imports.
- **Tests:** Unit tests were written against the v2 interfaces using in-memory
  fakes. The v1 stack had its own passing tests. Nothing tested the *relationship*
  between the two stacks.
- **CI:** Green across the board. The build does not know that two models are
  claiming to represent the same business domain.
- **Runtime:** Both stacks were registered on the router at different paths.
  The service started cleanly. No panics, no 500 errors during smoke tests.

The problem was an integrity concern, not a correctness concern â€” it required
reasoning about the system as a whole rather than evaluating any individual
component in isolation.

## Applied solution

The v1 stack was removed entirely:

- `LeatherProduct` domain type deleted.
- `LeatherProductRepository` interface and its PostgreSQL implementation deleted.
- `LeatherProductService` interface and `DefaultLeatherProductService` deleted.
- `LeatherProductHandler` and its route registration deleted.
- Migration `001_create_tables.sql` (`products` table) deleted; database connect
  function updated to only apply the v2 migration.
- Seed data referencing the `products` table removed.

The `stock_quantity` columns in `catalog_products` and `product_variants` were
retained as read-only snapshot fields for the dotCMS sync endpoint
(`GET /catalog/variant-inventory`), but all mutation paths for those fields were
removed. `inventory-service` was declared the single source of truth for stock,
and this was recorded explicitly in ADR-001 Â§5.

## Learning

**Every time a new domain model is introduced, perform an integrity analysis
before continuing to build on top of it.**

Concretely: when a second model is added to an existing domain (or when a
migration from model v1 to model v2 begins), schedule a checkpoint at the moment
the new model reaches feature parity. At that checkpoint, answer explicitly:

1. Is the old model still referenced anywhere in the routing, service, or
   repository layer?
2. Are there now two or more columns or services claiming to own the same
   business concept (e.g., stock quantity)?
3. Is the old model's migration still being applied, keeping an orphaned table
   in the schema?

If any answer is yes, resolve the conflict before adding new features. A
parallel stack is not scaffolding â€” it is divergence debt that compounds with
every subsequent commit.

This check should take at most an hour. Discovering it six weeks later, after
a second service and a frontend have been wired against the v2 model, is
measured in days.
