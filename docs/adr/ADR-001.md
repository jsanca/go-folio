# ADR-001: go-folio Architecture Decisions

**Status:** Accepted  
**Date:** 2026-05-27  
**Author:** Engineering

---

## Overview

This document records the key architectural decisions made during the design and
implementation of go-folio, a portfolio project modelling the backend and
frontend of a small leather-goods e-commerce store. Decisions are grouped by
concern; each entry explains the context, the choice made, and the trade-offs
accepted.

---

## 1. Monorepo with Go Workspace and Client/Server Split

### Context

A single leather-goods store needs a backend (several Go services) and a
frontend (a browser SPA). Keeping them in separate repositories would require
coordinating releases across repos and duplicating CI scaffolding.

### Decision

A single Git repository is used, with a hard physical boundary between backend
and frontend:

```
go-folio/
├── server/          ← Go workspace (catalog-service, inventory-service, gateway-service, gen)
├── client/          ← React SPA (Vite)
├── go.work          ← workspace root
└── docker-compose.yml
```

Go modules live under `server/` and are linked by a workspace `go.work` at the
repo root. The React project in `client/` is entirely independent — it has its
own `package.json` and knows nothing about Go.

### Rationale

- A single repo makes it trivial to keep proto definitions, seed data, and
  integration tests consistent across all services.
- The `server/` / `client/` split provides a clear boundary: a developer
  working only on the frontend never needs Go tooling, and vice versa.
- Go workspaces allow simultaneous development across modules (e.g., changing a
  type in `gen/` is immediately visible in `gateway-service/`) without
  publishing to a registry.
- Module import paths (`github.com/jsanca/go-folio/...`) are stable and do not
  change when the physical directory layout changes.

### Consequences

- `go build ./...` from the workspace root does not work (known Go workspace
  limitation: the pattern requires modules to be direct children of `.`). Each
  service is built from its own directory.
- The `buf.gen.yaml` output path must point to `server/gen/` so that proto
  regeneration writes to the correct location.

## 2. Three-Service Decomposition

### Context

The store needs at minimum: a product catalog, stock management, and a
client-facing API that combines both. A monolith would work, but this is a
portfolio project intended to demonstrate service decomposition and
inter-service communication patterns.

### Decision

Three services, each with a single responsibility:

| Service | Protocol | Port | Responsibility |
|---|---|---|---|
| `catalog-service` | REST / HTTP | 8080 | Products, variants, images, dotCMS sync |
| `inventory-service` | gRPC | 9090 | Stock levels, reservations, adjustments |
| `gateway-service` | REST / HTTP | 8090 | Client-facing aggregation |

### Rationale

- **Catalog** owns descriptive, relatively static data (name, price, images,
  color). It is read by the gateway and synced to dotCMS; REST/JSON is
  idiomatic for these use cases.
- **Inventory** owns mutable stock state with transactional semantics. gRPC is
  chosen here (see §3). The gateway calls it for every SKU on every product
  request; binary framing reduces per-call overhead.
- **Gateway** is the only surface the browser talks to. It composes catalog
  data + live stock into a single JSON response, shielding the client from
  internal protocol choices.

### Consequences

- Introducing a new product attribute requires only a catalog-service change; a
  stock-adjustment feature requires only an inventory-service change.
- The gateway is the single point where breaking changes in either downstream
  service need to be absorbed before reaching the client.

## 3. Protocol Choice: REST for Catalog and Gateway, gRPC for Inventory

### Context

Two protocol options were considered for service-to-service communication:
REST/JSON (HTTP/1.1) and gRPC (HTTP/2, Protocol Buffers).

### Decision

- **catalog-service** exposes a REST HTTP API.
- **inventory-service** exposes a gRPC API.
- **gateway-service** exposes a REST HTTP API (client-facing) and calls the
  others via their native protocols.

### Rationale

**Why REST for catalog-service:**
- The catalog's primary consumers are: (a) the gateway, which performs bulk
  reads, and (b) dotCMS, which uses cursor-paginated sync endpoints. Both are
  well-served by JSON over HTTP — the response payloads are large and
  human-readable, and the cursor pagination contract is naturally expressed as
  query parameters.
- REST is also simpler for the `GET /products` and `GET /catalog/variants/{sku}`
  endpoints added for gateway consumption.

**Why gRPC for inventory-service:**
- The gateway calls `GetStock` once per SKU per product-page request. With
  multiple variants per product, this is a high-frequency, low-payload RPC
  pattern where binary framing and HTTP/2 multiplexing reduce overhead.
- Protocol Buffers provide a versioned, strongly-typed contract (`inventory.proto`)
  that makes the stock data structure explicit and self-documenting.
- The streaming RPC (`SyncStock`) is a natural fit for future bulk
  reconciliation between catalog snapshot data and authoritative stock levels —
  something that would be awkward to express over REST.
- gRPC status codes (`NOT_FOUND`, `FAILED_PRECONDITION`, etc.) map cleanly to
  domain errors without needing to invent an error envelope convention.

**Why REST for gateway-service:**
- The gateway's clients are browsers and mobile apps. JSON/HTTP is universally
  supported; gRPC-Web requires additional infrastructure (Envoy proxy or
  grpc-web transpiling) that adds complexity without benefit for a portfolio
  project.

### Consequences

- The gateway must import both the `net/http` stack and the generated gRPC
  client stubs (`gen/inventory`), making it the only service with a mixed
  protocol dependency.
- On the inventory side, all client code must be regenerated whenever the proto
  changes (`buf generate`).

## 4. Transaction Ownership in the Service Layer (Unit of Work)

### Context

Inventory operations — particularly `ReserveStock` and `ReleaseStock` — require
atomic multi-row mutations (stock table + reservations table). The question is
whether transaction management should live in the repository layer or the
service layer.

### Decision

Transactions are owned and managed exclusively by the service layer.
Repositories expose a `WithTx(*sql.Tx) Repository` method that returns a
transaction-scoped copy of themselves; they never open transactions on their own.

```go
// Repository interface — repositories never open transactions themselves.
type Repository interface {
    HasAnyStock(ctx context.Context) (bool, error)
    GetStock(ctx context.Context, sku string) (*Stock, error)
    // ...
    WithTx(tx *sql.Tx) Repository
}

// Service owns the transaction lifecycle.
func (s *service) withTx(ctx context.Context, fn func(Repository) (T, error)) (T, error) {
    tx, err := s.db.BeginTx(ctx, nil)
    // ...
    result, err := fn(s.repo.WithTx(tx))
    // commit or rollback
}
```

The service holds a `*sql.DB` directly and calls `BeginTx` via a private helper
that handles commit/rollback uniformly.

### Rationale

- Business invariants (e.g., "available must not go negative") are expressed in
  the service layer. Placing transactions there keeps the read-modify-write cycle
  entirely within the layer that enforces those invariants.
- Repositories become pure data-access objects with no knowledge of whether they
  are executing inside or outside a transaction — the `querier` interface (common
  subset of `*sql.DB` and `*sql.Tx`) makes this transparent.
- Testing is straightforward: the fake repository used in unit tests implements
  `WithTx` as a no-op (returns `self`), while a real in-memory SQLite database
  is used to exercise the actual `BeginTx`/`Commit` lifecycle.

### Consequences

- Repository implementations must carry two execution paths — one backed by
  `*sql.DB` and one by `*sql.Tx`. The `querier` interface abstracts this at the
  cost of a small amount of boilerplate.
- Only inventory-service uses this pattern today. Catalog-service operations are
  currently single-row (no multi-table atomic writes needed), so it uses plain
  `*sql.DB` throughout.

## 5. Inventory as the Single Source of Truth for Stock

### Context

`ProductVariant` in catalog-service has `StockQuantity` and `StockStatus`
fields. The question was whether catalog-service should expose a mutation
endpoint to update those fields, or whether inventory-service should be the
sole authority on stock.

### Decision

`inventory-service` is the canonical source of truth for all stock data.
`ProductVariant.StockQuantity` and `ProductVariant.StockStatus` in
catalog-service are **read-only snapshot fields** — populated at seed time and
updated only by an explicit sync process (not yet implemented). No HTTP endpoint
in catalog-service mutates stock.

Specifically:
- `UpdateVariantInventory` was removed from `CatalogService`, `ProductVariantRepository`,
  and their implementations.
- The fields remain in the struct and the database schema for the dotCMS sync
  endpoint (`GET /catalog/variant-inventory`), which reads the cached values.
- The `StockStatus` that the client actually sees is derived live in the gateway
  from inventory-service data (see §6).

### Rationale

- Two services writing to stock creates race conditions and divergence.
  inventory-service already enforces non-negative quantities and handles
  reservation atomicity; duplicating that logic in catalog-service adds risk
  with no benefit.
- The snapshot fields in catalog serve the dotCMS sync use case: an external CMS
  needs a bulk export of catalog + rough stock info to display "Out of Stock"
  badges. This does not need to be real-time; a periodic sync from inventory is
  acceptable.

### Consequences

- The catalog snapshot may lag behind the real inventory by the sync interval.
  This is acceptable for CMS display purposes but not for checkout.
- Any future "add to cart" or "checkout" flow must call inventory-service
  directly (via the gateway), not rely on catalog-service stock fields.

## 6. StockStatus Derived at the Gateway, Not Stored

### Context

When the gateway aggregates a product response, it needs to expose a
`stockStatus` field (`IN_STOCK`, `LOW_STOCK`, `OUT_OF_STOCK`) to the client.
The options are: (a) store the status in inventory-service and return it via
`GetStock`, or (b) derive it in the gateway from the raw `available` count.

### Decision

The gateway derives `StockStatus` at request time from the `available` quantity
returned by `inventory-service.GetStock`. The threshold for `LOW_STOCK` is
configurable via the `LOW_STOCK_THRESHOLD` environment variable (default: 5).

```
available == 0           → OUT_OF_STOCK
0 < available ≤ threshold → LOW_STOCK
available > threshold    → IN_STOCK
```

### Rationale

- Status is a derived, presentational concept — a function of quantity and
  business policy. Storing it would require keeping it in sync with the
  quantity, adding a write on every stock adjustment.
- The threshold is a business parameter, not a data attribute. Making it
  configurable per deployment (via env var) without a schema change is cleaner.
- inventory-service stays domain-pure: it knows about counts, reservations, and
  adjustments, but not about the presentation layer's notion of "low stock".

### Consequences

- Different deployments (e.g., staging vs. production) can use different
  thresholds without a code change.
- If the threshold changes, existing clients immediately see updated statuses
  because nothing is cached.

## 7. Interface Segregation for CatalogService (Four Repository Interfaces)

### Context

The catalog domain touches four distinct concerns: products, variants, images,
and sync/pagination. All four are implemented by a single SQLite struct
(`SQLiteCatalogRepository`). The question is whether to expose one broad
interface or several narrow ones to `CatalogService`.

### Decision

`CatalogService` takes **four** separate repository interfaces as constructor
arguments:

```go
func NewCatalogService(
    products CatalogProductRepository,
    variants ProductVariantRepository,
    images   ProductImageRepository,
    syncRepo CatalogSyncRepository,
) CatalogService
```

`SQLiteCatalogRepository` satisfies all four. In the composition root
(`runtime.NewCatalogRuntime`), the same instance is passed for all four.

### Rationale

- Each test file only needs to implement the interfaces relevant to what it's
  testing. A test for product creation fakes only `CatalogProductRepository`; it
  does not need to implement sync pagination methods.
- Adding a new method to one interface does not force changes in fakes that do
  not use that interface.
- The explicit four-argument constructor makes each dependency visible at the
  call site — a reader of `NewCatalogService` can see exactly what data access
  the service performs.

### Consequences

- The composition root call looks redundant (`catalogRepo, catalogRepo, catalogRepo, catalogRepo`),
  but this is intentional and documented. The comment in `runtime.go` explains
  the trade-off.
- Adding a fifth concern would require a fifth interface and constructor
  parameter, which is a deliberate forcing function to keep the service focused.

## 8. Runtime / Composition Root Pattern (Thin `main.go`)

### Context

Go services often accumulate wiring logic in `main()`: connecting the database,
building repositories, creating services, registering routes. This makes `main`
hard to test and the startup sequence hard to understand.

### Decision

All three services follow the same startup pattern:

```
main.go → config.Load() → observability.NewLogger() → database.Connect()
        → runtime.New<Service>Runtime()
        → runtime.NewComposite()
        → seed.Run()
        → server.New().Start()
```

`runtime.New*Runtime()` is the **single wiring point** for each service. It
wires repositories into services (or clients into the gateway) and returns a
runtime struct whose fields are the public surfaces callers need. `main.go` stays
under ~40 lines and contains no business logic.

`CompositeRuntime` wraps one or more `io.Closer` objects and calls `Close()` on
each in reverse order, providing deterministic shutdown.

### Rationale

- `runtime/` packages are easily readable: one function, all dependencies
  explicit, no global state.
- Changing the wiring (e.g., swapping a SQLite repository for a Postgres one)
  requires modifying one function in one file.
- `CompositeRuntime` ensures gRPC connections, database handles, and other
  resources are released in the right order regardless of which service is
  shutting down.

### Consequences

- Services that require error-returning constructors (gateway-service, where
  `grpc.NewClient` can fail) return `(*Runtime, error)` instead of just
  `*Runtime`. `main.go` handles this as a fatal error.
- There is currently no graceful HTTP shutdown (no `signal.NotifyContext`). The
  process exits on `SIGKILL`. This is acceptable for a portfolio project but
  would need to be addressed before production use.

## 9. Observability Strategy

### Context

The services need enough observability to diagnose production issues without
over-engineering a portfolio project.

### Current State

| Concern | Implementation | Services |
|---|---|---|
| Structured logging | `log/slog` (JSON output) | All three |
| HTTP request logging | `observability.RequestLogger` middleware | catalog-service |
| Panic recovery | `observability.PanicRecovery` middleware | catalog-service |
| Prometheus metrics | `observability.NewMetrics` + `/metrics` endpoint | catalog-service only |
| Health / readiness | `/health` and `/ready` endpoints (Pinger interface) | catalog-service |

### Rationale

- **`log/slog`** (stdlib, Go 1.21+) was chosen over third-party loggers (zap,
  zerolog) to keep the dependency list short for a portfolio project. JSON
  output is compatible with log-aggregation stacks (Loki, Datadog, CloudWatch)
  without format conversion.
- **Prometheus** is the de-facto standard for Go service metrics. The middleware
  records HTTP method, path, status code, and latency. `/metrics` is scraped by
  Prometheus; Grafana can be added to docker-compose when dashboards are needed.
- **Health vs. Readiness** endpoints are kept separate: `/health` returns 200 as
  long as the process is running; `/ready` calls `Pinger.Ping()` (which does a
  `SELECT 1` on the database) and returns 503 if the DB is unreachable.

### Planned / Not Yet Implemented

- **Distributed tracing (OpenTelemetry):** The gateway makes two downstream
  calls per request; a trace-id should propagate from the gateway into
  catalog-service HTTP headers and the gRPC metadata of inventory-service calls.
  This would use `go.opentelemetry.io/otel` and an OTLP exporter.
- **Metrics for inventory-service and gateway-service:** Both services currently
  have no Prometheus instrumentation. inventory-service should expose gRPC
  request metrics; gateway-service should expose per-upstream latency histograms.
- **Centralized log correlation:** All services emit JSON logs independently.
  Adding a `trace_id` field to every log line (propagated from the gateway) is
  the minimal step needed for correlated log queries in production.

### Consequences

- The `Pinger` interface in `observability` keeps the health handler decoupled
  from `*sql.DB` — any object with `Ping(context.Context) error` can be used.
- Until distributed tracing is added, debugging a slow gateway response requires
  correlating logs across three separate log streams by timestamp.

## 10. SQLite as the Database for Catalog and Inventory

### Context

Both catalog-service and inventory-service need a relational database. The
candidates were SQLite and PostgreSQL.

### Decision

SQLite is used for both services. The driver is `github.com/mattn/go-sqlite3`
(CGO). Migrations are embedded SQL files applied at startup via
`database/embed.go`.

### Rationale

- SQLite requires zero infrastructure: the database is a single file, there is
  no separate process to start, and `docker compose up` needs no extra container.
  This is the right choice for a portfolio project where demonstrating the
  application logic matters more than demonstrating operational complexity.
- Embedding migrations ensures the schema is always up-to-date on startup
  without an external migration tool.
- The CGO dependency is contained: catalog-service and inventory-service use it;
  gateway-service has `CGO_ENABLED=0` in its Dockerfile and uses only pure-Go
  dependencies.

### Consequences

- CGO requires a C compiler at build time. The `golang:1.24-alpine` Docker image
  includes `gcc` via the Alpine toolchain.
- SQLite does not support horizontal scaling. Replacing it with PostgreSQL for
  production would require: changing the driver import, adjusting any
  SQLite-specific SQL (e.g., `INSERT OR IGNORE`), and updating the connection
  factory in `database/database.go`.
- Write concurrency in SQLite is limited to one writer at a time (WAL mode
  allows concurrent readers). This is acceptable for a single-instance portfolio
  service.

## 11. Keyset Cursor Pagination for Sync Endpoints

### Context

The catalog-service sync endpoints (`GET /catalog/product-projections`,
`GET /catalog/variant-inventory`) are designed for dotCMS bulk exports. The
result set may grow to thousands of rows; a reliable pagination strategy is
needed.

### Decision

Cursor-based keyset pagination is used. The cursor is a base64-encoded JSON
struct `{ updatedAt, id }`. Callers pass `?cursor=<token>&pageSize=<n>` on
subsequent requests.

```sql
WHERE (updated_at > ? OR (updated_at = ? AND id > ?))
ORDER BY updated_at ASC, id ASC
LIMIT ?
```

### Rationale

- **Offset pagination** (`LIMIT n OFFSET m`) is unstable: inserting or deleting
  rows between requests causes pages to skip or repeat records.
- The `(updatedAt, id)` keyset is stable because `updated_at` is monotonically
  non-decreasing for changed records, and `id` breaks ties deterministically.
- Encoding the cursor as opaque base64 JSON hides implementation details from
  callers and allows the cursor format to evolve without breaking the API
  contract.

### Known Limitation

`ListProductProjections` filters by `catalog_products.updated_at` only.
Variant-only or image-only changes are not surfaced unless the parent product
row is also touched. This is documented and accepted for the current scope; a
future improvement would use a view or a trigger to propagate child-table
`updated_at` values to the parent product.

### Consequences

- The `SyncToken` field in the response body captures the server-side timestamp
  at the start of the query. Callers can use it as the `updatedSince` filter in
  the next poll to avoid re-processing already-seen records.
- The dotCMS integration must store and forward the cursor; it cannot resume
  from a row number.

## 12. Gateway Fallback for Inventory Errors

### Context

The gateway calls inventory-service once per variant when building a product
response. If inventory-service is unavailable or returns `NOT_FOUND` for a SKU
(e.g., a variant exists in the catalog but has not been seeded in inventory),
the gateway must decide whether to fail the entire request or return partial
data.

### Decision

Any error from `inventory-service.GetStock` results in a graceful fallback:
`available: 0, reserved: 0, stockStatus: "OUT_OF_STOCK"`. A `NOT_FOUND` error
is silently swallowed; any other error is logged at `WARN` level but does not
propagate to the HTTP response.

### Rationale

- A catalog page that shows "Out of Stock" for a temporarily unreachable
  inventory service is far better than a 502 error that shows no products at
  all.
- `NOT_FOUND` specifically means the SKU is unknown to inventory — this is a
  data-consistency gap, not a transient error. Logging it at `WARN` would be
  noisy during development; silently returning the safe fallback is appropriate.
- Callers (the browser) see a complete product list at the cost of potentially
  stale stock info during an inventory outage.

### Consequences

- Misconfigurations (e.g., inventory-service never seeded a new SKU) are silent
  — the product will show "Out of Stock" with no visible error. This makes the
  seed data in `inventory-service/internal/seed/seed.go` important to keep in
  sync with the catalog.
- When distributed tracing (§9) is implemented, the fallback path should emit a
  span event so outages are visible in the trace waterfall even when HTTP
  responses look healthy.
